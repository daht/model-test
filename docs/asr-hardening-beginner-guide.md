# ASR 整改逻辑说明（入门版）

## 1. 这次要解决什么

当前项目可以把实时音频转换成文字，但“模型能识别”不等于“服务已经稳定”。实时识别过程中，模型会不断修改自己刚才输出的文字。例如：

```text
第一次：肯德基。
第二次：肯德基双人餐。
第三次：肯德基双人餐八件套兑换券。
```

前端必须知道哪些文字还可能变化，哪些文字已经最终确认。与此同时，GPU 推理比较慢，多个客户端同时说话时还会产生排队、超时和资源竞争。

这次整改不更换 Qwen3-ASR 模型，主要改造模型外围的服务逻辑，让它具备以下能力：

- 已经确认的文字不重复、不丢失、不再修改。
- 临时文字可以被模型正常修订。
- 一个客户端推理时，服务器仍能响应其他连接和健康检查。
- 服务器忙不过来时明确拒绝，不无限积压音频。
- 连接断开、超时或输入错误时能够正确清理模型状态。
- 所有行为都有自动测试和真实录音验收。

## 2. 需要先理解的几个名词

### ASR

Automatic Speech Recognition，自动语音识别。输入音频，输出文字。

### stateful 流式识别

客户端不断发送小块音频，模型保留前面音频和文字的状态，再结合新音频更新结果。它比“每两秒单独识别一次”更容易保持上下文，但状态管理也更复杂。

### partial

模型当前给出的临时文字。它可以在下一次结果中被替换。

```json
{"type":"partial","text":"肯德基双人"}
```

### sentence_final

服务器已经确认、以后不会修改的文字。客户端收到后永久追加。

```json
{"type":"sentence_final","text":"肯德基双人餐八件套兑换券。"}
```

### final

客户端结束录音时，服务器发送的最后一段未确认尾部。它不是整篇转写。

### VAD

Voice Activity Detection，语音活动检测。当前项目通过音量判断是否出现连续静音，静音达到阈值后提交文字。

## 3. 客户端应该怎样拼接文字

客户端始终按照下面的公式显示：

```text
所有 sentence_final 依次相加 + 最新一个 partial 或 final
```

例如服务器依次发送：

```text
partial:       肯德基双人餐
sentence_final: 肯德基双人餐八件套兑换券。
partial:       限时优惠
partial:       限时优惠仅需七十九元
final:         限时优惠仅需七十九元
```

最终显示：

```text
肯德基双人餐八件套兑换券。限时优惠仅需七十九元
```

`partial` 是替换，不是追加。如果把每个 partial 都追加，就会出现大量重复文字。

## 4. 为什么要单独做“文字状态机”

目前断句、累计文本、静音提交和 WebSocket 发送混在同一个 API 文件中。一个边界条件处理错误，就可能造成重复或丢字。

整改后增加一个独立的文字状态机。它只关心三件事：

```text
已经确认的前缀 confirmed_text
仍可修改的尾部 partial_text
正在观察的稳定标点 candidate
```

模型每返回一次累计文字，状态机负责计算新的 partial，以及是否应该产生 sentence_final。API 层不能再自行拼字符串。

状态机必须遵守：

- `confirmed_text` 只能越来越长。
- 每个 partial 都完整替换旧 partial。
- 每次提交后都发送最新 partial，即使它是空字符串。
- 如果模型突然修改已经确认的内容，停止当前会话并报告错误，不能把冲突文本重新追加。
- 每个事件带递增序号，方便排查是否遗漏或重复。

## 5. 为什么稳定时间要按音频计算

原实现使用服务器现实时间判断一个标点是否稳定。这个方式会受到网络速度影响。

假设客户端把 10 秒录音在 1 秒内快速上传：服务器现实时间只过去 1 秒，但模型实际上已经看到了 10 秒后续语音。反过来，如果网络停顿 1 秒，模型没有看到任何新音频，也不应该因此确认标点。

整改后使用：

```text
已处理音频秒数 = 已处理采样数 / 16000
```

只有模型继续处理了足够多的后续音频，并且标点仍然没有变化，才把文字确认下来。

## 6. 为什么需要“推理协调器”

Qwen/vLLM 的调用是同步操作。同步操作直接写在 FastAPI WebSocket 中时，GPU 推理期间整个事件循环可能被占住，其他连接连健康检查都无法及时响应。

整改后增加一个推理协调器：

```text
FastAPI 收到任务
       |
       v
放入有上限的推理队列
       |
       v
专用线程按顺序调用 Qwen/vLLM
       |
       v
异步通知 FastAPI 推理结果
```

模型加载、创建流式状态、增加音频、结束和清理全部在同一个专用线程中完成。这样可以避免多个线程同时修改同一个模型状态。

需要明确：协调器不会让一张 GPU 凭空变快。它的作用是保证服务不阻塞、状态不混乱、排队有上限。真实并发能力必须在 A10 上测量。

## 7. 服务器忙不过来怎么办

不能让客户端无限发送音频，然后几十秒后才收到所谓“实时”结果。

服务会限制：

- 同时在线的流式会话数量。
- 推理队列任务数量。
- 队列中累计的音频秒数。
- 每个音频包的最大字节数。
- 单连接允许落后实时音频的秒数。
- 单次连接时长和累计音频时长。

达到上限后，WebSocket 返回 `server_busy` 并使用关闭码 `1013`。客户端应稍后重试，而不是继续发送。

## 8. 为什么实时识别和文件识别要分开

一个长文件开始识别后，GPU 调用通常不能中途暂停。即使实时任务优先级更高，也只能等待长文件完成。

因此实时生产服务默认关闭文件识别：

```dotenv
ASR_FILE_TRANSCRIBE_ENABLED=false
```

文件识别最好部署成独立实例。只有一张 GPU 时，也至少要保证有实时连接期间不启动长文件任务。

## 9. 超时和断开连接为什么麻烦

Python 可以停止等待结果，但无法强行中止已经在 GPU 上运行的推理。

因此分成两种情况：

- 任务还在排队时超时：安全删除，模型没有被调用。
- 任务已经运行时超时：连接关闭，并把该会话标记为不可再使用；GPU 调用自然结束后再清理。

不能超时后继续复用原会话，否则迟到的 GPU 结果可能在后台修改状态，后续文字就会错乱。

## 10. 为什么一张 GPU 只能启动一个 ASR 进程

每个 Uvicorn worker 都会各自加载一份 Qwen/vLLM。如果配置 `--workers 2`，一张 GPU 上就可能出现两份模型，造成显存不足。

生产规则是：

```text
一张 GPU
一个 ASR 容器或进程
一个 Uvicorn worker
一份模型
```

需要更大并发时，应增加 GPU 和 ASR 实例，而不是在同一张卡上盲目增加 worker。

## 11. 健康检查和就绪检查的区别

`/health` 只回答“服务进程还活着吗”，必须快速返回。

`/ready` 回答“模型已经加载成功，而且现在可以接任务吗”。模型缺失、加载失败或协调器停止接单时，`/ready` 返回 503。

Docker 应使用 `/ready` 判断是否把服务当作可用实例。

## 12. 输入错误会怎样处理

以下输入不会再导致未处理异常：

- 非 JSON 的启动消息。
- `sample_rate` 不是数字或不是 16000。
- language 不是字符串。
- PCM 字节数是奇数。
- 单个音频包过大。
- 连接空闲或持续时间超过限制。

服务器会返回固定的错误代码和合适的 WebSocket 关闭码，客户端可以据此决定修正输入、重试或停止。

## 13. 实施分几个阶段

### 第一阶段：保证文字正确

- 抽离文字状态机。
- 使用音频时间判断稳定标点。
- 明确 partial、sentence_final 和 final 规则。
- 修复 chunked 模式吞重复词。

### 第二阶段：保证服务不会被拖死

- 增加单线程模型协调器。
- 增加队列、连接和音频时长限制。
- 处理超时、断开和会话清理。
- 将文件识别与实时流隔离。

### 第三阶段：保证可以上线和排障

- 增加 `/ready`。
- 增加不泄露音频和文本的性能日志。
- 更新客户端、Docker、配置和文档。
- 使用真实中文、日语录音回归。
- 在 A10 上测试 1、2、4、8 路并发。

## 14. 谁来实施和验收

只有收到明确的“可以执行”指令后才开始改代码。

执行顺序固定为：

1. 主 Agent 创建隔离工作区。
2. 单独的实施 Agent 按计划编码和运行开发测试。
3. 主 Agent 先审查实施改动。
4. 另一个独立测试 Agent 不依赖实施报告，从设计要求重新验收。
5. 测试发现问题后退回实施 Agent 修复，再由测试 Agent 复验。
6. 独立验收通过后，主 Agent 从完整产品角度回归 HTTP、WebSocket、脚本、配置、Docker 和真实录音流程。

实施 Agent 和测试 Agent 不能是同一个角色，避免自己实现、自己证明正确。

## 15. 什么情况下才能宣布完成

必须同时满足：

- ASR 单元测试和协议测试全部通过。
- 项目完整测试没有回归。
- 独立测试 Agent 给出通过结论。
- 主 Agent 的产品级 mock 流程通过。
- Docker/部署配置与文档一致。
- 真实 Qwen/vLLM 和 A10 验收通过。
- 没有剩余的已知范围内问题。

如果当前没有 GPU、模型权重或 Docker，只能明确报告哪些真实验收被阻塞，不能声称已经完成生产验收。
